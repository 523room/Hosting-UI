FD-01

Connector onboarding & activation

sequenceDiagram
    actor C as Connector
    participant CA as Connector App
    participant LP as Lead Platform
    participant DB as PostgreSQL
    participant EK as Setu OKYC · Aadhaar
    participant PN as Cashfree PAN
    participant PD as RazorpayX penny-drop
    actor CV as Converter
    participant CP as Converter Portal
    C->>CA: Mobile number + OTP (US-01)
    Note over CA,LP: ▶ SF-01 auth & session (Firebase phone OTP → JWT + session)
    LP->>DB: Upsert user (pending_activation)
    C->>CA: 5-step profile + onboarder code + consent (US-11, US-12)
    Note over CA,LP: ▶ SF-02 consent capture and ledger
    CA->>LP: POST /connector/onboard
    LP->>EK: Aadhaar OKYC (OTP)
    EK-->>LP: eKYC result
    LP->>PN: PAN verify — name + DOB match
    PN-->>LP: PAN match result
    LP->>PD: UPI penny-drop — name fetch
    PD-->>LP: Account-name match
    alt All three clean
    LP->>DB: status = active — auto-activated, no manual step
    else KYC exception / mismatch
    LP->>CV: Route to KYC-exception review queue (US-13)
    CV->>CP: Approve / reject with reason
    CP->>LP: Decision recorded
    end
    LP-->>CA: Push — activation result
Implementation notesOTP runs through a swappable OtpProvider adapter — first provider Firebase Auth phone method. Connector App calls POST /connector/onboard once profile + consent are captured; Lead Platform fires three named, independently swappable verifications synchronously — Aadhaar OKYC via Setu, PAN (name + DOB match) via Cashfree PAN, and UPI penny-drop name-fetch via RazorpayX. Clean across all three flips status to active in one write — no queue, no human in the loop. Any mismatch instead writes a row to the Converter's review queue table; the Converter's approve/reject is a second, separate write that the Connector App picks up on next poll/push.

FD-03

Qualification in under 90 seconds

sequenceDiagram
    participant CA as Connector App
    participant DB as PostgreSQL
    participant KYC as eKYC · PAN · Penny-drop
    actor CV as Converter
    participant TW as Temporal
    participant SETU as Setu (PAN + OKYC)
    participant CRIF as CRIF via Surepass
    actor B as Borrower
    TW->>SETU: PAN verify (US-27)
    alt PAN mismatch > 30%
    TW->>CV: Human review signal — never auto-reject (US-28)
    end
    TW->>B: Aadhaar OTP (OKYC)
    alt OKYC confirmed
    B-->>SETU: Confirmed (Aadhaar never stored) (US-29)
    else Retry fails twice
    TW->>CA: Fall back to physical KYC (US-29)
    end
    TW->>CRIF: Score-only soft pull — no footprint (US-30)
    TW->>DB: Score BAND only · 30-day purge scheduled (US-30)
    TW->>TW: Deterministic rules → Q001–Q007 (US-28)
    Note over TW: GenAI writes the reason text, never the decision
    TW-->>CA: QUALIFIED (+ commission estimate) or DISQUALIFIED (+ suggestion) (US-31, US-76)
Implementation notesThis whole flow runs inside a Temporal workflow so a crash mid-qualification resumes instead of restarting. PAN/OKYC calls hit Setu; CRIF is a soft pull via Surepass that returns a score only — the raw score never lands in the DB, only a derived band. The Q001–Q007 rule evaluation is deterministic code; the LLM is invoked only afterward, to phrase the reason, never to decide it.

SF-01

Auth & session (reusable)

sequenceDiagram
    actor U as User
    participant CL as Client (App / Portal)
    participant LP as Lead Platform
    participant DB as PostgreSQL
    U->>CL: Credentials — OTP (Firebase phone) or email+password (US-01)
    CL->>LP: POST /auth/verify
    opt FI portal / high-privilege
    LP-->>U: TOTP 2FA challenge (US-05a)
    U->>LP: TOTP code
    end
    LP->>DB: Verify + load role & fi_id tenant
    LP-->>CL: Short-lived access JWT + session token — 30-day cap, 12h idle (US-02)
    Note over LP: Deny-by-default RBAC + tenant scope (US-04)
    loop On refresh (re-arms 12h idle, up to 30-day cap)
    CL->>LP: Rotate session token (replay → revoke pair)
    end
Implementation notesOTP verification is a swappable adapter (first provider: Firebase Auth phone method). 2FA is conditional — only high-privilege clients (FI portal) get the TOTP challenge — but the token issuance path afterward is identical for every client. The session token carries a 30-day absolute cap with a 12-hour sliding idle expiry: each refresh re-arms the idle timer up to the hard cap, and a replayed (already-used) token revokes the whole pair, so theft is detected on first reuse. One active session per connector — a fresh login denylists the previous session's jti, so the same account cannot run on two devices at once. Access JWTs stay short-lived (15-minute TTL) and independent of session longevity. In plain terms: the app carries a disposable 15-minute pass that it silently swaps for a fresh one behind the scenes; the backend signs each pass and publishes a matching public "seal" (JWKS) so any service can check a pass is genuine without phoning home, and it periodically changes the lock (key rotation) so a leaked signing key has a short blast radius.

SF-07

RBAC & tenant scoping (reusable)

sequenceDiagram
    participant CL as Client
    participant MW as RequirePermission
    participant SC as Row-level scope
    participant DB as PostgreSQL
    CL->>MW: Request + JWT (role, fi_id) (US-04)
    alt Route not mapped to role
    MW-->>CL: 403 — permission denied (deny-by-default)
    else Permitted
    MW->>SC: role + fi_id
    Note over SC: connector→self · fi_*→own fi_id · onboarder→own-code · converter/billing/admin→GLOBAL
    alt Target out of remit / other tenant
    SC-->>CL: 404 — not found (existence hidden, never 403)
    else In remit
    SC->>DB: Scoped query
    DB-->>CL: Data — own remit only
    end
    end
Implementation notesTwo gates in series on every request. First, deny-by-default RequirePermission: any route not explicitly mapped to the caller's role returns 403. Second, a row-level scope filter binds the query to the caller's remit — connector self, fi_* own fi_id, onboarder own-code, while converter/billing/admin are truly global (all tenants, no geography/vertical scoping). One consistent rule governs the two error codes: 403 = action not permitted for your role; 404 = resource is real but outside your remit/tenant, so its existence is never disclosed. The 404-not-403 rule is applied identically to FI cross-tenant and connector cross-access.
